# Decision Model Evolution — Three-Phase Implementation Plan

## Purpose

Evolve StockWealth BD from a direct `BUY` recommendation model to a cautious,
conditional decision-support model without losing historical auditability or
breaking existing consumers.

The public decision labels will be:

- `POTENTIAL_BUY`
- `HOLD`
- `WAIT`
- `SELL`

`BUY` must disappear from user-facing surfaces, but not be blindly renamed or
removed from internal contracts. Existing immutable snapshots, replay reports,
legacy signals, comparisons, and API consumers may retain the legacy internal
meaning during a controlled compatibility period.

The desired outcome is **better conditional opportunity coverage**, not a
target quota of bullish results. A stock becomes `POTENTIAL_BUY` only when it
has completed, trustworthy data, passes all hard safeguards, and has an
actionable condition with a defined invalidation.

## Current-state findings and design constraints

The current implementation already has several foundations worth preserving:

- A canonical decision engine with eligibility, constraints, opportunity,
  risk, trade-plan, holder-action, and non-holder-action outputs.
- Immutable, versioned canonical decision snapshots and versioned cache input
  identity.
- Existing hard blocks for non-eligible data, corporate-action uncertainty,
  illiquidity, elevated trading risk, structural support failure, and an
  invalid fresh-entry plan.
- Existing replay/backtesting manifests, action/eligibility distribution
  monitoring, and constraint reason codes.

It also has two important limitations that define the implementation order:

1. The intraday `sync_market_snapshot` flow writes current-session rows into
   the daily-price source and rebuilds the universe. The canonical technical
   engine can therefore treat a partial-session row as a completed daily bar,
   including for volume averages, breakout detection, plans, regime inputs,
   cache identity, and immutable snapshots.
2. Opportunity quality and trade-plan feasibility are calculated separately,
   but a missing fixed overhead resistance target makes a fresh plan invalid.
   This is safe as a default, but it incorrectly excludes well-defined
   breakout/continuation cases in a bullish trend.

The sequence below fixes data/session truth before expanding plan semantics or
changing public terminology.

## Shared model rules

### Internal and display taxonomy

Use separate, versioned fields:

```text
internal_action              legacy/canonical decision semantics, initially BUY/HOLD/WAIT/SELL
display_action               POTENTIAL_BUY/HOLD/WAIT/SELL
entry_timing                 READY/PULLBACK/BREAKOUT/CONTINUATION/null
decision_taxonomy_version    v2
```

Initial compatibility mapping:

```text
internal BUY + valid READY plan -> display POTENTIAL_BUY + READY
internal HOLD/WAIT/SELL       -> mapped only through the new explicit resolver
```

Do not map every historical or current `BUY` mechanically. A v2
`POTENTIAL_BUY` must be generated by the new eligibility and plan resolver.
Historical records retain their stored taxonomy/version and their original
meaning.

### Public versus holder-aware action

Base analysis remains deterministic and user-independent. Holder-aware wording
is resolved only where a confirmed holding context exists.

| Context | Intact actionable opportunity | No actionable entry | Broken structure |
| --- | --- | --- | --- |
| Non-holder | `POTENTIAL_BUY` with condition | `WAIT` | `SELL` or contextual `AVOID` |
| Holder | `HOLD` with risk/management guidance | `HOLD` or `REVIEW` guidance | `SELL` |

The generic stock surface must not claim `HOLD` merely because a hypothetical
holder could hold it. It should use the deterministic display decision and
explain the appropriate non-holder condition.

### Hard safeguards

No market regime, display mapping, or conditional plan may bypass:

- completed, trustworthy decision data;
- eligibility and corporate-action safeguards;
- acceptable liquidity and tradability;
- valid structural or ATR-based invalidation;
- risk and extension limits;
- manipulation/suspicion and severe gap-risk safeguards; or
- a meaningful, testable entry condition.

## Phase 1 — Canonical session integrity and observability

### Goal

Make a completed DSE session the only source for canonical decisions,
snapshots, caches, market regime, and replay. Preserve intraday information as
a distinct provisional/live layer.

### Scope

1. Define one authoritative `decision_session_date`: the latest completed DSE
   trading session, not merely the latest ingested row.
2. Change canonical decision/universe input selection so it excludes the
   current open session. Apply the same date to:
   - daily-price windows and technical indicators;
   - volume baselines and breakout confirmation;
   - trade plans and recommendations;
   - market-summary/regime inputs;
   - universe cache identity and persisted canonical snapshots; and
   - backtest/replay observation construction.
3. Keep the current intraday price, turnover, DSEX/breadth, and freshness UI,
   but classify it explicitly as provisional. It must not alter a canonical
   EOD decision while the session is open.
4. Add clear API/read-model metadata:

   ```text
   decision_session_date
   live_data_as_of
   is_live_session
   ```

5. Ensure an after-close/final ingestion path explicitly finalizes the session
   before it is eligible for canonical rebuild and snapshot persistence.
6. Add an operator decision funnel that aggregates existing constraint codes.
   It must reconcile from `total_universe` to final displayed actions and show
   at least:

   ```text
   eligible
   bullish_opportunity
   hold / wait / sell
   blocked_by_data / liquidity / extension / entry_plan / risk
   ```

### Implementation boundaries

- Prefer a read-path/session-finality guard first; do not rush into a broad
  daily-price schema rewrite if the existing session scheduler can safely
  select the prior completed session.
- If the current model cannot express finality unambiguously, add the smallest
  persistence marker necessary (for example a session-finality state or a
  dedicated live snapshot record). Do not overwrite immutable decision
  history.
- Regime rows must be filtered to the same `decision_session_date` as stock
  inputs. A live index summary cannot be mixed with prior-session stock bars.
- Do not publish or persist a canonical snapshot for an open session.

### Primary affected areas

- `jobs/ingestion/ingest_daily_market_prices.py`
- `jobs/market_session_schedule.py` and session validation
- market-data repository freshness/window queries
- `modules/market_universe/market_universe_service.py`
- `modules/market_universe/market_universe_compute.py`
- `decision/market_regime.py`, canonical input lineage, cache/snapshot write
  paths, and replay builders
- API schemas, frontend freshness/live-context types, operations documentation

### Acceptance tests

1. A partial intraday row cannot replace the last completed row used for a
   canonical decision, volume baseline, breakout, plan, or snapshot.
2. A live resistance touch does not qualify as completed-session breakout
   confirmation.
3. Canonical universe, stock detail, regime, cache identity, and snapshot all
   agree on `decision_session_date`.
4. Intraday UI still exposes its own `live_data_as_of` and `is_live_session`.
5. Funnel counts reconcile exactly with the final universe rows.

### Phase gate

Do not begin new plan types or public taxonomy rollout until a current open
session can no longer affect canonical decisions or replay inputs.

## Phase 2 — Conditional opportunity and entry-plan model

### Goal

Make opportunity quality, entry readiness, and entry timing explicit, while
preserving hard risk safeguards. This phase creates the information required
for meaningful `POTENTIAL_BUY` decisions; it does not lower thresholds to make
the count larger.

### Canonical outputs

Add typed, versioned outputs to the decision result:

```text
opportunity_quality   e.g. weak / constructive / strong, with existing score retained
entry_readiness       not_ready / conditional / ready
entry_timing          READY / PULLBACK / BREAKOUT / CONTINUATION / null
blocker_codes         ordered, machine-readable constraint reasons
```

Keep the current opportunity score, evidence strength, risk label, eligibility,
constraints, and trade-plan status for continuity. Do not encode the new
meaning only in a free-text explanation.

### Decision resolver

The resolver must follow this ordering:

1. Apply data, eligibility, corporate-action, liquidity, risk, extension, and
   structural-invalidation hard blocks.
2. Assess opportunity quality from completed-session evidence.
3. Determine whether a concrete plan is valid, then determine its timing.
4. Produce `POTENTIAL_BUY` only for a strong bullish opportunity with a valid
   plan and a meaningful trigger/condition. Otherwise produce `WAIT` (or
   deterministic bearish `SELL`), not an artificial potential buy.

| Completed-session result | Public decision | Entry timing |
| --- | --- | --- |
| Strong opportunity + executable valid plan | `POTENTIAL_BUY` | `READY` |
| Strong opportunity + supported pullback zone | `POTENTIAL_BUY` | `PULLBACK` |
| Strong opportunity + confirmed/conditional breakout rule | `POTENTIAL_BUY` | `BREAKOUT` |
| Strong opportunity + narrow managed continuation policy | `POTENTIAL_BUY` | `CONTINUATION` |
| Bullish evidence but no safe actionable plan | `WAIT` | null |
| Failed hard safeguard | `WAIT` or `SELL` as appropriate | null |

### Plan types

Implement each plan as a focused domain result, not frontend branching.

#### `READY`

Requires a completed-session entry zone, invalidation/stop, acceptable initial
risk and extension, and either a practical objective or an approved management
policy. User wording: “Entry setup is currently available near the shown
range.”

#### `PULLBACK`

Requires a strong opportunity and a precise support, moving-average, breakout
retest, or structural entry zone. Expose `preferred_entry_zone`,
`invalidation_price`, `condition_text`, and an expiry/reassessment rule.

#### `BREAKOUT`

Requires a clear trigger/resistance, a completed-session close/confirmation
rule where required, participation confirmation, invalidation below the
breakout structure, and expiry. An intraday touch is never confirmation.
Expose `trigger_price`, `confirmation_rule`, `invalidation_price`, and
`condition_text`.

#### `CONTINUATION`

Start with one deliberately narrow policy for a completed breakout or
price-discovery situation with no reliable overhead resistance. It requires a
structural or ATR stop, initial-risk cap, extension control, trailing rule,
time/reassessment stop, and optional partial-profit checkpoint.

Support target/management modes:

```text
STRUCTURAL / ATR_PROJECTION / MEASURED_MOVE / TRAILING
```

Unknown upside is not infinite reward. A trailing plan must not claim a fixed
risk/reward ratio that it cannot substantiate.

### Controlled regime integration

Refine market regime after session integrity is complete. Keep its influence
centralized and capped through a single regime result:

```text
regime_score
regime_label
regime_phase       EARLY / HEALTHY / EXTENDED / REVERSAL_RISK
regime_confidence
```

Initially permit it only to influence candidate ranking, relative-strength
priority, explanation/confidence context, and whether an otherwise-valid
breakout/continuation plan type is permitted. It must not change hard
safeguards or introduce scattered hidden score bonuses.

### Primary affected areas

- `decision/recommendation.py`, `constraints.py`, `trade_plan.py`, `engine.py`
- opportunity/risk/evidence/technical result types and trading constants
- canonical decision, stock-detail, universe, scanner, and market-pulse
  schemas
- regime service and replay observation model

### Acceptance tests

1. Strong bullish evidence without a valid plan remains `WAIT`.
2. Hard data, liquidity, corporate-action, gap-risk, or invalidation failures
   cannot yield `POTENTIAL_BUY`.
3. Each of `READY`, `PULLBACK`, `BREAKOUT`, and `CONTINUATION` produces a
   valid, explainable condition and invalidation.
4. A no-overhead-resistance continuation uses the managed/trailing policy, not
   an artificial fixed target.
5. Regime modifiers cannot bypass hard constraints.

### Phase gate

Replay the new outputs on completed, point-in-time sessions. Do not change
numeric thresholds merely because `POTENTIAL_BUY` coverage is low or high.

## Phase 3 — Public taxonomy rollout, compatibility, and validation

### Goal

Expose the simplified public decision model without breaking current clients,
historical snapshots, signals, comparisons, or replay interpretation. Then use
the new taxonomy to observe and validate behavior.

### Compatibility rollout

1. Add `internal_action`, `display_action`, `entry_timing`, and
   `decision_taxonomy_version` to new v2 canonical/API read models.
2. Preserve legacy `TraderRecommendation.BUY` and legacy action fields during
   the compatibility window. Map legacy internal `BUY` to public
   `POTENTIAL_BUY` only at the versioned display resolver.
3. Keep immutable historical snapshot payloads and their original
   `action_taxonomy` unchanged. New snapshot identities must carry v2 taxonomy
   metadata; never rewrite old rows.
4. Version cache keys/envelopes and persisted decision comparisons so old and
   new semantic values are never compared as if identical.
5. Move public frontend badges, labels, filters, scanner wording, dashboard
   actionability, and watchlist displays to `display_action`. Display the
   entry condition next to `POTENTIAL_BUY`, not as a hidden detail.
6. Derive holder-specific actions in the watchlist/user-context layer; retain
   deterministic base analysis for all generic stock endpoints.

### Affected consumer review

Review and update deliberately rather than performing an enum rename:

- stock-details and canonical-decision APIs;
- market universe and scanner filters/ranking;
- signals and legacy signal mappings;
- dashboard and market-pulse presentation;
- watchlist holder/non-holder resolver;
- persisted signal comparisons and `NEW` badge logic;
- frontend API types, signal badges, action filters, and tests;
- cache payloads and immutable snapshot persistence;
- backtesting metrics, reporting, and documentation.

### Replay and operational validation

Replace the single `canonical_buy` reporting lens with separate cohorts:

```text
POTENTIAL_BUY + READY
POTENTIAL_BUY + PULLBACK
POTENTIAL_BUY + BREAKOUT
POTENTIAL_BUY + CONTINUATION
```

Use only completed, point-in-time sessions. Report 3, 5, 10, and 20-session
outcomes by cohort, liquidity bucket, regime label, and regime phase:

- trigger activation and expired-without-entry rates for conditional plans;
- hit rate, average/median return, expectancy, and false-breakout rate;
- maximum adverse/favourable excursion and drawdown;
- costs/slippage sensitivity where data supports it; and
- action/funnel/blocker distributions.

Treat these as observation and validation outputs first. Threshold tuning is a
separate approved experiment with a frozen holdout period; it is not part of
this implementation unless a structural threshold is required for correctness.

### Documentation and migration requirements

- Add a database migration only if new persistence is required for session
  finality, live snapshots, or new stored canonical fields. Do not migrate
  historical values in place.
- Update `backend/docs/api_collection.md` for every API shape change.
- Update decision-support and operations docs with public taxonomy, internal
  timing, session rules, compatibility behavior, funnel semantics, and replay
  interpretation.
- Bump strategy/threshold/input/taxonomy versions only according to the
  existing versioning rules; distinguish output-taxonomy changes from input
  schema changes.

### Acceptance tests

1. Legacy internal `BUY` resolves to public `POTENTIAL_BUY` through the v2
   compatibility layer.
2. All four entry timings use the same public badge and show their condition.
3. Holder and non-holder actions resolve correctly without making the generic
   base decision user-dependent.
4. Historical snapshots, persisted comparisons, and `NEW` badges remain
   backward-compatible.
5. API/cache versioning prevents semantic cross-version comparison.
6. Replay uses completed, point-in-time sessions and reports all entry-timing
   cohorts separately.

## Delivery sequence and review checkpoints

Implement in small, independently reviewable changes. At the start of each
phase, inspect the actual callers and contracts rather than assuming this plan
is exhaustive.

1. Phase 1: session selection/finality, live metadata, canonical cache and
   snapshot protections, and funnel baseline.
2. Phase 2: typed opportunity/readiness/timing model; narrow conditional plans;
   centralized regime result; replay-only observation.
3. Phase 3: v2 display taxonomy, frontend and API compatibility rollout,
   holder-aware presentation, and taxonomy-aware replay/reporting.

Each change set should include focused tests, API documentation updates, and a
backward-compatibility note. At the end of delivery, report changed files,
migrations/commands, test results, compatibility status, and remaining replay
or threshold-tuning work.
